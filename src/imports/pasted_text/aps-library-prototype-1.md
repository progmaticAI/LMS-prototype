Prototype Requirements

Create a fully connected prototype where:

Every sidebar item opens its own page.
Every table row opens a detailed record.
Every "Add" button opens a complete form.
Every "Edit" button opens an edit screen.
Every workflow contains multiple connected screens.
Every dialog exists.
Every CRUD operation is represented.
Include loading states, empty states, confirmation dialogs, error states, and success notifications.
Include realistic sample data for books, students, teachers, transactions, and reports.
Use modern enterprise components such as advanced data tables, filters, breadcrumbs, search, pagination, bulk actions, dashboards, charts, and contextual menus.
Final Goal

Generate two complete, enterprise-grade, interconnected prototypes:

APS Headquarters Admin Portal (for monitoring and administration across all 14 schools).
Individual School Library Management System (the day-to-day operational software used by librarians).

The School Library Management System should contain the majority of the prototype (around 70–80% of the screens) because it is where all daily library operations occur. The Headquarters Portal should focus on oversight, analytics, reporting, branch management, and system administration. Both applications should share a consistent design system while clearly reflecting their different purposes and user roles. The final prototype should be comprehensive enough to present to stakeholders and serve as a blueprint for development.
